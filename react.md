# Deep Tree

## Введение
Deep Tree — файловая модульная архитектура React.js приложений, ориентированная на масштабируемость больших фронтенд-проектов.

Архитектура основана на принципах [SOLID](https://en.wikipedia.org/wiki/SOLID), адаптированных для организации файлов фронтенд-проектов.

## Цели
- Установить единую структуру файлов в проекте
- Улучшить читаемость и навигацию по коду
- Разбивать большие файлы на понятные части
- Снизить технический долг проекта
- Повысить гибкость и переиспользование кода
- Снизить порог входа для новых разработчиков

## Введение в сущности
Сущность — это программная логическая единица.

### Зачем нужны сущности?

Сущность — это кирпич архитектуры. Без атомарных единиц нет из чего строить. Каждый файл или папка имеет чёткий смысл, и ничего лишнего в одной единице не хранится.

К примеру могут быть такие виды сущностей: 
component, style, image, hook, function, page, section, route, 
helper, config, type, api-client, translation и тд.

Сущность всегда отвечает за что-то одно. (принцип единой ответственности SOLID).

> Некоторые сущности могут быть уникальны и в одном экземпляре. Например: config или mui-theme.

В архитектуре глубокого дерева система упорядочивает **сущности** в виде дерева, 
где каждая сущность имеет доступ к родительским и управляет своими дочерними сущностями.

Любая сущность может быть как _отдельным файлом_, так и _группой файлов, объединённых папкой_.




### Составная сущность
Составная сущность — это группа файлов, объединённых общей папкой. Внутри могут быть слои (см. слои), файловые слои (см. файловые-слои) и любые вспомогательные файлы/папки, необходимые для работы.

#### Зачем нужна составная сущность?

Составная сущность и её `index.ts` решают проблему закрытости. Одиночный файл в любой момент можно превратить в папку с несколькими файлами — и внешний путь импорта не изменится. Снаружи никто не знает о внутреннем устройстве: это защита от случайного вмешательства в детали реализации.

#### Правила составной сущности
- Имя папки — PascalCase.
- Обязателен корневой `index.ts`; экспорт наружу — только через него.
- Не вкладывать другие составные сущности того же уровня (только слои/файловые-слои и одиночные сущности).
- Структура внутри папки самодостаточна.

`index.ts` — только реэкспорт, никакой логики:
```ts
// HomePage/index.ts
export { HomePage } from './HomePage';
```






### Одиночная сущность
Одиночной (атомарной) сущностью считается файл, 
который самодостаточно хранит один смысловой объект: 
код (класс, функцию/хук, конфигурацию), данные, текстовое описание, изображение, шрифт, видео и т.п.

#### Правила одиночной сущности
- Внутри только один самостоятельный смысловой объект (класс, функция, конфигурация, один набор данных, одно изображение и т.п.).






## Базовые сущности
Визуально любой макет можно разделить на три условных уровня компонентов:
- **Страницы (pages / views)**
- **Секции (sections)**
- **Компоненты (components)**

### Страницы
Страницы — крупные React-компоненты, составные сущности. Это логически независимые части приложения; каждая выполняет свою бизнес‑роль и не повторяется.

#### Зачем нужны страницы?

Страницы делят приложение на крупные независимые куски. Каждая страница — отдельный контекст с понятной бизнес-ролью. Это не даёт приложению превратиться в монолит, где всё перемешано.

#### Правила страниц
- Всегда составная сущность с именем в PascalCase и суффиксом `{NAME}Page` (пример: HomePage).
- Обязателен корневой `index.ts`; экспорт наружу — только через него.
- Обязателен файл представления `${DIR_NAME}.tsx` (DIR_NAME = имя папки страницы).
- Обычно без props.
- Хранятся только в слое `pages`.

#### Исключения правил страниц
- На страницу может приходить верхнеуровневый скоуп данных (например, локаль, словарь переводов, 
конфиг feature‑флагов, ключи интеграций), если это касается всего экрана.

#### Примеры страниц
- HomePage — начальная страница, корневая,
- AboutPage — рассказывает о компании или продукте, знакомит с командой, событиями и тд
- CatalogPage — представления продуктов, фильтрация, поиск, сортировка
- ContactUsPage — способы связи

Минимальная структура страницы:

```
src/
  pages/
    HomePage/
      index.ts            # единая точка экспорта
      HomePage.tsx        # представление страницы
      sections/
        HeroSct/
          index.ts
          HeroSct.tsx
        BenefitsSct/
          index.ts
          BenefitsSct.tsx
      components/
        CallToAction/
          index.ts
          CallToAction.tsx
```







### Секции
Секции — зависимые части родительского **React-компонента**, составные сущности; дробят его логику на понятные блоки.

#### Зачем нужны секции?

Секции спасают от раздутых компонентов. Без них вся логика страницы или крупного компонента оседает в одном файле. Секция забирает часть этой логики в свою папку — родительский компонент остаётся читаемым, а каждый блок разбирается отдельно.

> На секции всегда можно поделить любой другой компонент. Page, section, component могут иметь секции!


#### Правила секции
- Всегда составная сущность с именем в PascalCase и суффиксом `{NAME}Sct` (пример: HeroSct).
- Обязателен корневой `index.ts`; экспорт наружу — только через него.
- Обязателен файл представления `${DIR_NAME}.tsx` (DIR_NAME = имя папки секции).
- Props не передаём.
- Живёт только в слое `sections`; может содержать слои, кроме `pages`.
- Рекомендация: Не создавайте слой `sections` ради одной секции — оставляйте логику в родителе или выносите повторяемые части в `components` до появления хотя бы двух секций.

#### Исключение правил секции
- В редких случаях в секцию может попадать пропс, но, если хочется переиспользовать секцию целиком, это сигнал вынести повторяемые части в компоненты.

#### Примеры секции
К примеру HomePage слишком большая, если попытаться создать страницу целиком — код будет слишком большим.
Естественно большой код это малая гибкость, сложное внедрение и плохая читаемость.

Как мы помним, любая страница разделена на логические секции, логично разделить ее на секции.  
Условные:
- HeroSct
- BenefitsSct
- FeaturesSct
- ContactUsSct
Каждая секция — неотъемлемая часть логики HomePage, и каждая секция независима от другой секции. 
В данном примере каждая секция имеет в конце приставку Section или сокращенно Sct.

Минимальная структура секции:

```
src/
  pages/
    HomePage/
      sections/
        FeaturesSct/
          index.ts          # экспорт секции
          FeaturesSct.tsx   # представление секции
          components/
            FeatureCard/
              index.ts
              FeatureCard.tsx
          hooks.ts          # файловый слой для локальной логики секции
```

Секция не принимает props — данные берёт из контекста или локального хука:
```tsx
// FeaturesSct/FeaturesSct.tsx
import { useHomeContext } from '../../context';
import { FeatureCard } from './components/FeatureCard';

export function FeaturesSct() {
  const { features } = useHomeContext();

  return (
    <section>
      {features.map(f => <FeatureCard key={f.id} {...f} />)}
    </section>
  );
}
```








### Компоненты
Компоненты — повторяемые React-компоненты (составные сущности), которые можно использовать в разных секциях и страницах.

#### Зачем нужны компоненты?

Компоненты — это то, что повторяется. Кнопка, карточка, аватар — одно и то же в разных местах. В отличие от секции, компонент не привязан к конкретному родителю: его задача именно в повторении и переиспользовании.

> Компоненты хранятся в слое **components**.
> Для разделения логики **компонента** могут быть использованы секции.

#### Правила компонентов
- Всегда составная сущность с именем в PascalCase.
- Обязателен корневой `index.ts`; экспорт наружу — только через него.
- Обязателен файл представления `${DIR_NAME}.tsx` (DIR_NAME = имя папки компонента).
- Хранятся только в слое `components`.
- Могут содержать слои, кроме `pages`.

Минимальная структура компонента:

```
src/
  components/
    Avatar/
      index.ts          # экспорт компонента
      Avatar.tsx        # представление
      types.ts          # файловый слой с типами, если нужно
      styles.module.scss
```



Рекомендуемая схема именования (мягкое правило): основной файл совпадает с именем сущности (SomeComponent.tsx), 
рядом лежат вспомогательные без префикса
(types.ts, styles.module.scss, index.ts),
а тесты/сторис повторяют имя сущности (SomeComponent.test.ts,
SomeComponent.stories.ts). 
Держите эту схему последовательной внутри слоя, чтобы поиск и навигация оставались предсказуемыми.

## Слои
Слой — это отдельная папка‑пространство имён, в которой логически группируются сущности.
Название папки и есть имя слоя: например components, hooks, sections, pages и т.п.
Если появляется новый тип сущности, необходимо создать для него свою папку‑слой.

### Зачем нужны слои?

Слои создают рамки и контекст разработки. Зашёл в `hooks` — там хуки и правила работы с ними. Зашёл в `managers` — там менеджеры со своей логикой. Слой сигнализирует, что здесь можно ожидать, и не даёт разным типам сущностей перемешиваться в одном месте.

### Правила слоев:
- Имя слоя — camelCase.
- Слой не вкладывается напрямую в слой: новый слой появляется только после сущности (у компонента может быть свой `sections`, у секции — свой `components` и т.д.).
- Слой может содержать группы.
- Внутри только сущности своего типа (хуки в `hooks`, компоненты в `components`, стили в `styles`).
- `index.ts` может присутствовать, но не обязателен.

### Примеры использования слоев:
- components(layer)->Button(entity). - ok
- components(layer)->useGenPass(entity). - not ok
- hooks(layer)->other(group)->useSomeHook(entity). - ok
- components(layer)->Button(entity)->hooks(layer)->useOnMouseMoveHandler(entity) - ok

### Пример названия слоев:
- pages
- sections
- components
- ui
- hooks
- utils
- types
- api
- styles

## Файловые слои (single-file layers)
Файловые слои — временно выполняющий роль слоя, когда кода мало и нет вложенных сущностей или групп. 
Со временем он должен быть развёрнут в полноценную папку слоя при росте функционала.

### Когда можно
- Вся логика слоя умещается в один файл и остаётся читаемой.
- Нет вложенных сущностей/групп; это временная форма слоя.

### Когда разворачивать в обычный слой
- Появляется вторая сущность того же типа или файл становится неудобен для навигации/тестов.
- Появляется потребность во вложенной структуре (группы). Обычно из за количества сущностей внутри файлового слоя.

### Примеры имен инлайновых слоёв
- `types.ts`
- `hooks.ts`
- `utils.ts`
- `helpers.ts`
- `constants.ts`
- `styles.module.scss`

### Правила инлайновых слоев
- Имя файла совпадает с типом слоя; префиксы не нужны.
- Не смешивайте разные типы сущностей в одном инлайновом слое (hooks и utils в одном файле — нельзя).
- При разворачивании сохраняйте имя слоя и разложите содержимое по отдельным сущностям.


## Группы
Группы — это сущности, объединённые общим смыслом в одном слое или другой группе.
Группа может быть разделена на подгруппы, а те на другие группы и так до бесконечности по необходимости.
Задача группы разгружать слои (см. слои)

### Зачем нужны группы?

При масштабировании слои разрастаются и превращаются в свалку. Группы наводят порядок внутри слоя — без изменения структуры самого дерева. Это защита от хаоса на уровне одного слоя при большом количестве сущностей.

### Правила групп
- Имя группы — camelCase.
- Слои внутри группы не допускаются; группы могут вкладываться в группы.
- Внутри только сущности типа слоя-родителя (в `components` — только компоненты).
- Группа не служит контейнером для одной-единственной сущности.
- Имя группы не совпадает с именами слоёв.
- `index.ts` может присутствовать, но не обязателен.

### Пример использования групп:
- components(layer)->buttons(group)->Button(entity) =  ок
- components(layer)->buttons(group)->outlined(group)->Button(component) = ок
- others(group) -> components(layer) -> Button(entity) = NOT ок
- components(layer)->buttons(group)->useSomeHook(entity) = NOT ок
- components(layer)->buttons(group)->Button(entity)->hooks->someHooks(group)->useSomeHook(entity) = ок

### Пример групп:
- **others** (ex: hooks(layer)->**others**(group))
- **header** (ex: components(layer)->**header**(group))
- **report** -> (ex: components(layer)->**report**(group))


## Шеринг сущностей (sharing)
Дочерняя сущность может использовать слои и файловые слои всех своих предков вплоть до корня `src`.
Чем глубже сущность, тем больше доступных зависимостей сверху.
Сущность существует только в той ветке дерева, где она реально используется, и не распространяется на соседние ветки родителя.

### Зачем нужен шеринг?

Шеринг — это постоянный живой процесс в проекте, а не разовое решение. Он решает проблему веса кода: чем выше сущность в дереве, тем больше у неё зависимостей и ответственности; чем ниже — тем она локальнее. Появилась задача, увидели что компонент начал повторяться в разных местах — подняли его выше в ближайший общий предок. Это и есть шеринг.

### Правило доступа
Импортировать можно только **вверх по своей ветке** — из родительских слоёв, файловых слоёв и одиночных сущностей.
Импорт из **соседней ветки** не допускается.

Файлы **представления** (`.tsx`) предка — не источник импорта для детей.
Если нужно пробросить что-то вниз по дереву, выносите это в отдельную одиночную сущность или файловый слой рядом с представлением:

| Что шарим | Как оформить |
|-----------|--------------|
| Контекст | `context.ts` — одиночная сущность рядом с `.tsx` |
| Общие стили | `styles.module.scss` — файловый слой рядом с `.tsx` |

```
HomePage/
  index.ts
  HomePage.tsx              # представление — дети не импортируют напрямую
  HomePage.module.scss      # стили самого HomePage
  context.ts                # одиночная сущность — доступна всем детям
  styles.module.scss        # файловый слой — общие стили для детей
  sections/
    HeroSct/
      HeroSct.tsx           # ← может импортировать context.ts и styles.module.scss
```

`context.ts` — определяет контекст и хук доступа к нему:
```ts
// HomePage/context.ts
import { createContext, useContext } from 'react';

type THomeContext = {
  title: string;
  features: TFeature[];
};

export const HomeContext = createContext<THomeContext | null>(null);

export function useHomeContext() {
  const ctx = useContext(HomeContext);
  if (!ctx) throw new Error('useHomeContext: нет провайдера');
  return ctx;
}
```

### Пример

```
src/
  components/              ← доступно всем
    Button/
  pages/
    HomePage/
      components/          ← доступно HomePage и всем её детям
        CallToAction/
      sections/
        HeroSct/           ← видит: src/components, HomePage/components
          components/      ← доступно только внутри HeroSct
            Highlight/     ← видит: src/components, HomePage/components, HeroSct/components
        FeaturesSct/       ← видит: src/components, HomePage/components
                           ← НЕ видит: HeroSct/components (соседняя ветка)
```

### Когда поднимать сущность выше
Если сущность нужна в двух и более ветках — переносите её в ближайший общий слой-предок.

Пример: `HeroSct` и `FeaturesSct` оба используют `FeatureCard` → выносите в `HomePage/components`.

## Глубина вложенности проекта
Глубина **вложений не ограничивается,** если это локальный, крупный функционал (например, тяжёлое модальное окно).
Ориентир: углубляйтесь столько, сколько нужно для читаемости внутри этого контекста;
поднимать сущности выше имеет смысл только при реальном шаринге за пределами текущей ветки.

### Зачем нужна глубина вложенности?

Глубина решает проблему фокуса. Чем глубже работаешь — тем меньше кода вокруг затрагиваешь, тем точнее и локальнее изменение. Спустился в нужное место, поправил только там, где нужно, и пошёл дальше. Поднимать сущности выше — только когда они реально нужны в нескольких местах, а не "на всякий случай".


## Менеджер
Менеджер — составная независимая сущность: вынесенный провайдер с контекстом и логикой,
сфокусированный на одной конкретной задаче.

По сути — более лёгкая альтернатива глобальному стейт-менеджеру: не требует зависимостей,
не хранит всё в одном месте, синтаксически свободен.

### Правила менеджера
- Всегда составная сущность с именем в PascalCase и суффиксом `{NAME}Mng` (пример: ReportMng).
- Обязателен корневой `index.ts`; экспорт наружу — только через него.
- Обязателен файл контроллера `useController.ts` — передаваемая логика хранится в нём.
- Обязателен файл представления `${DIR_NAME}.tsx` (DIR_NAME = имя папки менеджера).
- Живёт только в слое `managers`.
- Зачастую не использует никаких props кроме `children`.

### Файловая структура менеджера

```
src/
  managers/
    ReportMng/
      index.ts          # единая точка экспорта
      ReportMng.tsx     # компонент-обёртка / провайдер
      useController.ts  # хук с бизнес-логикой и стейтом внутри
```

`useController.ts` — вся бизнес-логика и стейт менеджера:
```ts
// ReportMng/useController.ts
import { useState } from 'react';

export function useController() {
  const [items, setItems] = useState<TReportItem[]>([]);

  function addItem(item: TReportItem) {
    setItems(prev => [...prev, item]);
  }

  return { items, addItem };
}
```

`ReportMng.tsx` — провайдер, пробрасывает результат контроллера в контекст:
```tsx
// ReportMng/ReportMng.tsx
import { createContext, useContext, ReactNode } from 'react';
import { useController } from './useController';

type TReportMng = ReturnType<typeof useController>;

const ReportContext = createContext<TReportMng | null>(null);

export function useReportMng() {
  const ctx = useContext(ReportContext);
  if (!ctx) throw new Error('useReportMng: нет провайдера');
  return ctx;
}

export function ReportMng({ children }: { children: ReactNode }) {
  const controller = useController();
  return (
    <ReportContext.Provider value={controller}>
      {children}
    </ReportContext.Provider>
  );
}
```

### Когда использовать менеджер

Менеджер — всегда составная сущность. Если логика слишком проста — это хук или `context.ts`, а не менеджер.

| Ситуация | Решение |
|----------|---------|
| Стейт нужен только в одном компоненте | Локальный `useState` / `useReducer` |
| Нужно пробросить данные без drilling, логика простая | `context.ts` + хук у родителя |
| Есть бизнес-логика, несколько источников вызова | Менеджер |
| Нужен UI над деревом (тосты, модалки, оверлеи) | Менеджер с `renderOwn` |
| Нужен time-travel, продвинутые devtools | Redux / Zustand |

Обычный `context` решает только задачу props-drilling — у него нет структуры, правил и выделенного места в проекте. Менеджер — это составная сущность со своими слоями и конвенциями.

Глобальный стейт-менеджер (Redux/Zustand) делает данные доступными везде по умолчанию. Менеджер физически недоступен там, где нет его провайдера — это не договорённость, это ограничение самого React.

`AuthPage` не может случайно взять `useCartMng` — провайдера там просто нет.

**Пример:** `useTheme` с простым toggle + persistence — достаточно хука или `context.ts` в `App`.
Если появляется синхронизация с системной темой, анимации переходов, несколько тем — оправдан `ThemeMng`.

> Один вариант менеджера — всегда составная сущность. Не нужно думать "какой тип выбрать".

### Рекомендуемый стек
Менеджер хорошо работает в связке с **react-query**:
- **react-query** — серверный стейт: загрузка, кэш, синхронизация, оптимистичные обновления.
- **Менеджер** — локальный UI-стейт: тема, плеер, модалки, корзина.

В большинстве случаев этой связки достаточно. Redux/Zustand остаётся актуальным там, где нужны продвинутые devtools и time-travel отладка — но это редкий, а не дефолтный сценарий.

### Компоненты внутри менеджера

Менеджер по природе своей — центр притяжения логики. Со временем внутри могут появиться хуки, утилиты, API-клиенты и даже компоненты. Это не нарушение архитектуры, но требует осознанного подхода.

**Ключевой вопрос: кто вызывает и кто рендерит.**

Если компонент рендерится **самим провайдером** — он принадлежит менеджеру:

```tsx
export function ReportMng({ children }: { children: ReactNode }) {
  const controller = useController();
  return (
    <ReportContext.Provider value={controller}>
      {children}
      {controller.isOpen && <ReportModal />}  {/* менеджер владеет своим UI */}
    </ReportContext.Provider>
  );
}
```

Это стандартный паттерн для toast-менеджеров, оверлеев и глобальных модалок — UI должен быть выше дерева потребителей.

Если же компонент используется **потребителями** менеджера — это сигнал вынести его в `components`.

---

**Как принять решение:**

| Ситуация | Решение |
|----------|---------|
| UI рендерится провайдером, одно место вызова | Локальный контекст или стейт — менеджер избыточен |
| UI рендерится провайдером, несколько мест вызова | Компонент внутри менеджера — провайдер рендерит сам |
| Компонент нужен потребителям извне | Вынести в слой `components` |

> **Пример из практики:** кнопка в хедере вызывает метод менеджера → всплывает модалка. Если это единственное место вызова — локальный контекст проще и чище. Если таких мест два и больше — менеджер с `<ReportModal />` внутри провайдера становится правильным выбором.



## Утилиты (опционально)

Паттерны контекста и менеджера содержат повторяющийся бойлерплейт.
Deep Tree предлагает две утилиты, чтобы его устранить.

### `createSafeContext`

Создаёт контекст и безопасный хук доступа к нему в одну строку.

```ts
// utils/createSafeContext.ts
import { createContext, useContext } from 'react';

export function createSafeContext<T>(displayName: string) {
  const Context = createContext<T | null>(null);
  Context.displayName = displayName;

  function useCtx(): T {
    const value = useContext(Context);
    if (!value) throw new Error(`${displayName}: нет провайдера`);
    return value;
  }

  return [Context, useCtx] as const;
}
```

Без утилиты:
```ts
// HomePage/context.ts
export const HomeContext = createContext<THomeContext | null>(null);

export function useHomeContext() {
  const ctx = useContext(HomeContext);
  if (!ctx) throw new Error('useHomeContext: нет провайдера');
  return ctx;
}
```

С утилитой:
```ts
// HomePage/context.ts
export const [HomeContext, useHomeContext] = createSafeContext<THomeContext>('HomeContext');
```

---

### `createManager`

Создаёт провайдер менеджера и хук доступа к нему — весь шаблонный код менеджера исчезает.

Принимает опциональный `renderOwn` — для случаев когда провайдер должен рендерить собственный UI (модалки, тосты, оверлеи).

```tsx
// utils/createManager.tsx
import { ReactNode } from 'react';
import { createSafeContext } from './createSafeContext';

export function createManager<T>(
  useController: () => T,
  renderOwn?: (value: T) => ReactNode,
  displayName?: string,
) {
  const [Context, useCtx] = createSafeContext<T>(displayName ?? 'Manager');

  function Provider({ children }: { children: ReactNode }) {
    const value = useController();
    return (
      <Context.Provider value={value}>
        {children}
        {renderOwn?.(value)}
      </Context.Provider>
    );
  }

  return [Provider, useCtx] as const;
}
```

Без утилиты:
```tsx
// ReportMng/ReportMng.tsx
const ReportContext = createContext<TReportMng | null>(null);

export function useReportMng() {
  const ctx = useContext(ReportContext);
  if (!ctx) throw new Error('useReportMng: нет провайдера');
  return ctx;
}

export function ReportMng({ children }: { children: ReactNode }) {
  const controller = useController();
  return (
    <ReportContext.Provider value={controller}>
      {children}
    </ReportContext.Provider>
  );
}
```

С утилитой — простой менеджер:
```tsx
// ReportMng/ReportMng.tsx
export const [ReportMng, useReportMng] = createManager(useController, undefined, 'ReportMng');
```

С утилитой — менеджер с собственным UI:
```tsx
// ReportMng/ReportMng.tsx
export const [ReportMng, useReportMng] = createManager(
  useController,
  (ctrl) => ctrl.isOpen && <ReportModal />,
  'ReportMng'
);
```



## Типы данных
Deep Tree рекомендует максимально строгий подход к типизации.

При создании типа или интерфейса используется префикс `T` — для единообразия и читаемости.
Например: TProfile, TItem, TEntity, TTeamMember и тд.


### Правила типов данных
- Имя всегда начинается с префикса `T` (Type) и пишется в PascalCase.
- Для `interface` используется тот же префикс `T`.

### Примеры названия типов
**TTeamMember**
**TProfile**
